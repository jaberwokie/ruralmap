## Why this happens

Lovable generates project thumbnails by snapshotting the published/preview URL. The thumbnail can break for reasons unrelated to app code:

1. **Snapshot failed/expired** — Lovable's screenshot service couldn't capture the page (timeout while Leaflet tiles loaded, transient error). The cached thumbnail URL now 404s.
2. **Map-heavy first paint** — The app's initial paint is a Leaflet map that fetches external tiles (CartoDB) and large GeoJSON. If the snapshot fires before tiles render, or the snapshotter blocks third-party requests, the capture fails.
3. **Custom domain / publish state mismatch** — Project has a custom domain (`ruralmap.opsframe.io`) plus the lovable.app URL. If the thumbnail points at a stale URL, it breaks.
4. **OG image is unrelated** — `public/og-image.jpg` controls social previews (Twitter/Facebook), not the Lovable dashboard thumbnail. So that file being correct doesn't help here.

## What this is NOT

- Not a code bug — the app runs fine (verified: no console/runtime errors, map renders).
- Not an OG image issue — dashboard thumbnails are server-generated screenshots, not OG metadata.
- Not something a code change in the repo can directly fix.

## Fix path (no code changes)

The dashboard thumbnail is regenerated by Lovable's infrastructure, not by editing the repo. The reliable ways to force a refresh:

1. **Re-publish the project** — Click Publish (top-right web icon on desktop, or `…` menu → Publish on mobile). A fresh deploy typically triggers a new thumbnail snapshot.
2. **Make any small visible edit and re-publish** — Forces the snapshot pipeline to re-run.
3. **Hard refresh the dashboard** — The broken image may just be a stale browser cache of the thumbnail URL. Cmd/Ctrl+Shift+R on the projects list.
4. **Wait** — Lovable retries thumbnail generation periodically; transient failures often self-heal within a few hours.

If after re-publish + hard refresh the thumbnail is still broken, it's a platform-side issue with the screenshot service for this specific project and should be reported to Lovable support — no repo change will fix it.
